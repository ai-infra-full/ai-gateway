# LLM Gateway 实现方案

## 设计原则

在 AI 辅助开发环境下，代码生成和维护成本已经显著降低，因此没有必要为了减少少量重复代码而设计复杂的统一抽象层。

本项目采用简单、直接的实现方式：

* 分别实现 OpenAI，Anthropic 和 Gemini 协议的 Rust 服务端接口。
* 每种协议使用独立的数据结构和处理流程。
* 不强行将所有请求转换成统一的 Canonical Request IR。
* 公共能力沉淀到共享库中。
* Provider层负责对接具体的后端LLM。
* 客户端使用什么协议请求，Gateway就使用相同协议返回。
* 代码验证，采用OpenAI，Anthropic 和 Gemini的官方client SDK(golang版本)验证。

核心目标不是消除所有重复代码，而是降低协议转换复杂度，保持类型清晰，易于验证正确性，并尽可能完整地兼容各家厂商的原生能力。
---

## 输入层

Gateway分别暴露三套原生协议接口。

### OpenAI

支持：

* POST /v1/chat/completions
* POST /v1/responses
* POST /v1/embeddings
* GET  /v1/models

### Anthropic

支持：

* POST /v1/messages
* POST /v1/messages/count_tokens
* POST /v1/messages/batches

### Gemini

支持：

* POST /v1beta/models/{model}:generateContent
* POST /v1beta/models/{model}:streamGenerateContent
* POST /v1beta/models/{model}:countTokens

每种输入协议对应独立的 Rust Handler 和请求类型。

```text
OpenAI API ───────► OpenAI Handler
Anthropic API ────► Anthropic Handler
Gemini API ───────► Gemini Handler
```

不要求三种请求首先转换成完全一致的统一数据模型。不支持的接口一律直接透传到支持的后端LLM.

---

## 公共库(暂时不用实现)

公共库只承载真正通用的能力，不负责抹平所有协议差异。

可以包含：

* 身份认证
* API Key 管理
* 模型注册表
* 路由选择
* 供应商选择
* 限流
* 超时控制
* 重试
* 熔断
* 请求追踪
* 日志
* Metrics
* Token 统计
* 成本统计
* 内容安全检查
* 请求上下文
* 错误分类

公共库提供基础能力，但不定义一个包含所有厂商字段的超大型请求结构。

## Provider 层

Provider层负责连接真正的后端LLM服务。

```text
providers/
├── openai/
├── anthropic/
└── gemini/
```

每个 Provider 独立实现：

* 请求构造
* HTTP 调用
* 身份认证
* 错误解析
* 流式数据解析
* Tool Calling映射
* Usage 统计解析
* Provider特有参数处理

Provider层可以暴露少量公共Trait，但Trait只描述调用生命周期，不要求所有协议共享完全相同的请求结构。

例如：

```rust
pub trait Provider<Req, Resp> {
    async fn invoke(
        &self,
        request: Req,
        context: &RequestContext,
    ) -> Result<Resp, ProviderError>;
}
```

不同 Provider 可以使用不同的 `Req` 和 `Resp` 类型。

不建议定义类似下面这种过度统一的数据结构：

```rust
struct UniversalLlmRequest {
    messages: Vec<UniversalMessage>,
    tools: Vec<UniversalTool>,
    reasoning: Option<UniversalReasoning>,
    response_format: Option<UniversalResponseFormat>,
    // 不断增加各种供应商专有字段
}
```

这种设计最终通常会演变成：

* 大量 `Option` 字段
* 大量枚举嵌套
* Provider 特有扩展字段
* 复杂的双向协议转换
* 原生能力丢失
* Streaming Event 难以准确映射

## 路由和策略层

路由、策略和校验逻辑位于协议 Handler 与 Provider Adapter 之间。

```text
Client Request
      │
      ▼
Protocol Handler
      │
      ▼
Routing / Policy / Validation
      │
      ▼
Provider Adapter
```

该层负责：

* 根据模型名选择 Provider
* 根据租户策略选择后端
* 模型别名解析
* 故障转移
* 请求限流
* 参数校验
* 权限校验
* 请求审计
* 成本限制
* 上下文长度限制

该层操作的是请求上下文和路由元数据，不必理解所有协议字段。

例如：

```rust
pub struct RouteRequest<'a> {
    pub protocol: Protocol,
    pub requested_model: &'a str,
    pub tenant_id: &'a str,
    pub streaming: bool,
}
```

路由结果可以是：

```rust
pub struct RouteDecision {
    pub provider: ProviderId,
    pub upstream_model: String,
    pub timeout: Duration,
    pub retry_policy: RetryPolicy,
}
```

## 输出策略

Gateway 采用协议对称原则：

> 输入是什么协议，输出就是什么协议。

具体表现为：

```text
OpenAI Chat Request
        │
        ▼
OpenAI Chat Response
```

```text
OpenAI Responses Request
        │
        ▼
OpenAI Responses Event
```

```text
Anthropic Messages Request
        │
        ▼
Anthropic Messages Response
```

```text
Gemini GenerateContent Request
        │
        ▼
Gemini GenerateContent Response
```

即使底层实际调用的是其他 Provider，最终也需要返回客户端所使用协议对应的响应格式。

例如：

```text
Anthropic Request
      │
      ▼
Routing
      │
      ▼
Gemini Provider
      │
      ▼
Anthropic Response
```

这里可以存在必要的协议转换，但转换发生在明确的协议边界中，而不是所有请求都先进入一个庞大的全局 Canonical IR。

## 推荐架构

```text
                         Client Request
                               │
             ┌─────────────────┼─────────────────┐
             ▼                 ▼                 ▼
       OpenAI Handler    Anthropic Handler   Gemini Handler
             │                 │                 │
             └─────────────────┼─────────────────┘
                               ▼
                    Routing / Policy / Validation
                               │
                               ▼
                       Provider Selection
                               │
                               ▼
                       Provider Adapter
                  ┌────────────┼────────────┐
                  ▼            ▼            ▼
               OpenAI       Anthropic      Gemini
                  │            │            │
                  └────────────┼────────────┘
                               ▼
                     Protocol Response Mapper
                               │
             ┌─────────────────┼────────────────────┐
             ▼                 ▼                    ▼
     OpenAI Chat Response  Anthropic Response  Gemini Response
             │
             ▼
     OpenAI Responses Event
```

其中：

* Handler 负责解析客户端协议。
* Core 负责公共能力。
* Routing 层负责模型和 Provider 选择。
* Provider Adapter 负责调用后端。
* Response Mapper 负责恢复客户端期望的协议格式。

这样做虽然会产生一定重复代码，但具有以下优势：

* 转换逻辑容易查找。
* 数据流清晰。
* 单个协议升级不会影响所有模块。
* 可以针对不同协议编写独立测试。
* 容易保留 Provider 特有字段。
* Streaming 转换更加可控。

## 是否需要 Canonical IR

完全统一的 Canonical IR 并不是必需的。

可以只设计一个较小的内部公共模型，用于承载真正通用的信息，例如：

```rust
pub struct CommonRequestMetadata {
    pub request_id: String,
    pub tenant_id: String,
    pub requested_model: String,
    pub stream: bool,
}
```

Tool、Message、Content Block、Reasoning、Cache Control 和 Streaming Event 等协议实体，仍保留在各自协议模块中。

也可以针对特定场景设计局部 IR。

例如，普通文本对话可以转换成：

```rust
pub struct BasicConversation {
    pub system: Option<String>,
    pub messages: Vec<BasicMessage>,
}
```

但对于以下复杂功能，可以继续使用协议专用模型：

* 多模态输入
* Tool Calling
* Reasoning
* Prompt Cache
* Computer Use
* Structured Output
* Server-Sent Events
* Provider 特有 Metadata

因此更合适的策略是：

> 使用最小公共模型，而不是追求完整统一模型。

## 方案优势

这种架构的主要优势是：

1. 协议边界清晰。

2. Rust 类型能够直接对应厂商协议。

3. 不需要维护庞大的 Canonical IR。

4. 新增厂商特有字段时，不必修改全局模型。

5. Streaming Event 可以按协议独立处理。

6. 更容易实现较高程度的原生 API 兼容。

7. AI 可以快速生成重复的 Adapter 和 Mapping 代码，重复代码的成本已经低于复杂抽象的维护成本。

8. 当某个 Provider API 发生变化时，影响范围相对有限。

---

## 需要接受的成本

该方案也需要接受一定代价：

* 协议转换代码数量较多。
* 同一种逻辑可能在多个 Handler 中重复。
* 不同 Provider 之间需要编写显式映射。
* 测试组合数量会增加。
* 跨协议转发时，部分特有能力仍然无法做到完全等价。

不过这些成本是显式且可控的。

相比之下，过度统一的抽象通常会将复杂度隐藏在：

* 大量枚举
* 动态字段
* 扩展属性
* 条件分支
* 不透明的序列化逻辑
* 难以理解的 Streaming 状态机

因此，在 AI 辅助编码场景中，选择适当重复、显式映射和低抽象架构，通常更加符合长期维护需求。

---

## 结论

本项目不追求通过一个统一接口完全抽象 OpenAI、Anthropic 和 Gemini。

推荐采用以下策略：

* 三套原生协议入口。
* 三套独立 Rust Handler。
* 公共能力放入共享库。
* Provider 独立实现。
* 协议转换显式完成。
* 输入协议与输出协议保持一致。
* 只抽象稳定且真正通用的能力。
* 允许一定程度的重复代码。
* 避免构建过于庞大的 Canonical IR。

整体设计原则可以概括为：

> 协议保持原生，公共能力复用，转换逻辑显式，抽象保持最小。
